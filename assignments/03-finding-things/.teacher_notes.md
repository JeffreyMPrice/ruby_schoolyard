# Teacher Notes — Assignment 03: Finding Things

## What the Learner Needs to Do

Implement seven class methods on `Product`, each using a specific ActiveRecord
query method. The expected implementations are:

| Method | Implementation |
|--------|---------------|
| `fetch(id)` | `find(id)` |
| `search_by_name(name)` | `find_by(name: name)` |
| `affordable(max_price)` | `where("price <= ?", max_price)` |
| `sorted_by_price` | `order(price: :asc)` |
| `top_priced(n)` | `order(price: :desc).limit(n)` |
| `all_names` | `pluck(:name)` |
| `all_ids` | `ids` |

## Paired specs

`fetch` has 2 specs (returns record + raises on miss) and `search_by_name` has
2 specs (returns match + returns nil). Each pair goes green simultaneously when
the method is implemented — this is intentional. Each method is one unit of work
and both specs together document its complete behaviour.

## Common student mistakes

- **Using `find_by(id:)` instead of `find`** for `fetch`. Functionally similar
  for the happy-path spec, but `find_by` returns nil on miss rather than raising.
  The second spec catches this.
- **Missing the `n` parameter on `top_priced`**. Some students hardcode a number
  instead of using the argument in `limit(n)`.
- **Returning a Relation instead of an Array for `all_names`/`all_ids`**. The
  specs use `match_array` / `contain_exactly` which work on both, so this won't
  cause failures — but it's worth discussing `pluck` vs `map`.
- **Sorting direction**. `sorted_by_price` spec uses `eq([cheap, mid, pricey])`
  (order-sensitive) so ascending must be explicit. Students who omit `:asc`
  still get ascending by default on most databases, but it's good practice to
  be explicit.

## Spec count

9 specs total — all on Product.

No SQLite-specific limitations in this assignment. All query methods used are
standard across databases.
