# Teacher Notes — Assignment 04: Products and Categories

## What the Learner Needs to Do

Add three association declarations — one per model — to make all 5 specs pass.

| File | Line to add | Spec(s) unlocked |
|------|------------|-----------------|
| `app/models/product.rb` | `belongs_to :category` | `product#category` returns the right record |
| `app/models/category.rb` | `has_many :products, dependent: :destroy` | `category#products` returns associated products; `dependent: :destroy` deletes products on category destroy |
| `app/models/review.rb` | `belongs_to :product, optional: true` | `where.associated(:product)` and `where.missing(:product)` query specs |

## Intentional pairs / groups

- Adding `belongs_to :category` to Product makes **one** spec green: `product.category`.
  Clean 1-for-1.

- Adding `has_many :products, dependent: :destroy` to Category makes **two** specs green:
  `category.products` and the cascade-destroy spec. These are a natural pair — both
  require `has_many` and neither can be meaningfully separated from it. `dependent:
  :destroy` is a single keyword argument on the same declaration.

  If you want maximum TDD granularity, you can ask the student to add `has_many
  :products` first (one spec green), then add `, dependent: :destroy` (one more green).
  The specs are written to support this — the cascade spec is separate from the
  collection spec.

- Adding `belongs_to :product, optional: true` to Review makes **two** specs green:
  `where.associated(:product)` and `where.missing(:product)`. Both depend on the same
  single declaration, and both are on the Review model — isolated from the
  Product/Category association.

## Why product_id on Review is nullable

Reviews start with `product_id: nil` (by design). This lets the `where.missing`
spec find orphaned reviews without any extra setup. If `product_id` were non-nullable,
the concept of "missing" association could not be demonstrated. Because `product_id`
is nullable, the student must use `optional: true` — otherwise Rails 5+ validation
would reject any review without a product.

## Why category_id on Product is nullable

Products start with `category_id: nil` to allow the product factory to work without
requiring a category. Rails 5+ `belongs_to` validation does NOT fire for `category_id`
here because there is no `optional: true` — yet the saves still work. This is a subtle
point: Rails validates the association at the model layer (checking the object reference),
not the database layer. When you do `create(:product)` via FactoryBot without setting
`category_id`, Rails accepts it because no `belongs_to :category` is declared yet
(that's what the student adds). Once the student adds `belongs_to :category`, the
factory still works because `create(:product, category: nil)` does not trigger
the presence validation — only `create(:product, category_id: nil)` with an explicit
nil assignment would. Students may add `optional: true` out of caution; it doesn't
break anything but is unnecessary here.

## Common student mistakes

- **Forgetting `dependent: :destroy`** — the cascade spec is specific enough to
  catch this: students who write only `has_many :products` will see the count
  assertion fail.
- **Forgetting `optional: true` on Review** — the most likely mistake. Without it,
  `create(:review)` in the `where.missing` spec will fail with a validation error
  rather than a query mismatch. The spec failure message names `optional: true`
  explicitly to guide them.
- **Confusing which model gets `belongs_to`** — some students put it on Category.
  The spec failure message names the model explicitly to guide them.

## Spec count

5 specs total:
- 1 on Product (`#category` accessor)
- 2 on Category (`#products` collection, `dependent: :destroy` cascade)
- 2 on Review (`where.associated(:product)`, `where.missing(:product)`)

## SQLite notes

No SQLite-specific limitations in this assignment. `where.associated` and
`where.missing` generate standard `INNER JOIN` / `LEFT OUTER JOIN ... WHERE IS NULL`
SQL that works identically across all databases.
