# Teacher Notes — Assignment 04: Products and Categories

## What the Learner Needs to Do

Add two association declarations — one per model — to make all 5 specs pass.

| File | Line to add | Spec(s) unlocked |
|------|------------|-----------------|
| `app/models/product.rb` | `belongs_to :category` | `product#category` returns the right record; `where.associated` and `where.missing` query specs |
| `app/models/category.rb` | `has_many :products, dependent: :destroy` | `category#products` returns associated products; `dependent: :destroy` deletes products on category destroy |

## Intentional pairs / groups

- Adding `belongs_to :category` to Product makes **three** specs green simultaneously:
  `product.category`, `where.associated(:category)`, and `where.missing(:category)`.
  This is intentional — all three depend on the same single declaration, and splitting
  them would require artificial extra steps. The README and spec failure messages make
  clear that all three are unlocked by the same line.

- Adding `has_many :products, dependent: :destroy` makes **two** specs green:
  `category.products` and the cascade-destroy spec. These are a natural pair — both
  require `has_many` and neither can be meaningfully separated from it. `dependent:
  :destroy` is a single keyword argument on the same declaration.

  If you want maximum TDD granularity, you can ask the student to add `has_many
  :products` first (one spec green), then add `, dependent: :destroy` (one more green).
  The specs are written to support this — the cascade spec is separate from the
  collection spec.

## Why category_id is nullable

Products start with `category_id: nil` (by design). This lets the `where.missing`
spec find uncategorised products without any extra setup. If `category_id` were
non-nullable with a foreign key constraint, every product would require a category
and the concept of "missing" association could not be demonstrated.

## Common student mistakes

- **Forgetting `dependent: :destroy`** — the cascade spec is specific enough to
  catch this: students who write only `has_many :products` will see the count
  assertion fail.
- **Writing `belongs_to :category, optional: true`** — not needed here. Rails 5+
  makes `belongs_to` required by default, but since `category_id` is nullable, the
  model will still save. Students may add `optional: true` out of caution; it doesn't
  break anything but is unnecessary.
- **Confusing which model gets `belongs_to`** — some students put it on Category.
  The spec failure message names the model explicitly to guide them.

## Spec count

5 specs total:
- 3 on Product (`#category`, `where.associated`, `where.missing`)
- 2 on Category (`#products` collection, `dependent: :destroy` cascade)

## SQLite notes

No SQLite-specific limitations in this assignment. `where.associated` and
`where.missing` generate standard `INNER JOIN` / `LEFT OUTER JOIN ... WHERE IS NULL`
SQL that works identically across all databases.
