# Teacher Notes — Assignment 02: Keeping Data Clean

## What the Learner Needs to Do

### Product model (`app/models/product.rb`)

1. **Presence validations**
   - `validates :name, presence: true`
   - `validates :price, presence: true`

2. **Numericality validations**
   - `validates :price, numericality: { greater_than: 0 }`
   - `validates :stock_count, numericality: { only_integer: true, greater_than_or_equal_to: 0 }`

3. **Custom validator** — description must differ from the product name (case-sensitive
   equality, both strings taken as-is). The spec checks for the exact message
   `"must differ from the product name"` on `errors[:description]`:
   ```ruby
   validate :description_differs_from_name

   private

   def description_differs_from_name
     return unless name.present? && description.present?
     if description == name
       errors.add(:description, "must differ from the product name")
     end
   end
   ```

### Category model (`app/models/category.rb`)

1. `validates :name, presence: true`
2. `validates :name, uniqueness: true`

## Spec count

9 specs total: 7 on Product, 2 on Category.

## Common student mistakes

- **Forgetting `return unless` guards in custom validators.** If the guard is missing and
  `name` or `description` is nil, the comparison crashes with `NoMethodError` rather than
  just being skipped.
- **Case-sensitive vs case-insensitive uniqueness.** The spec uses the default
  `validate_uniqueness_of(:name)` which is case-sensitive. Students sometimes add
  `case_sensitive: false` — that would make the uniqueness spec fail. The DB index
  (created in the migration) is also case-sensitive by default on SQLite.
- **Combining presence and numericality on price.** Both are needed: `presence: true`
  catches `nil`; `numericality: { greater_than: 0 }` catches zero and negatives. Some
  students try to use numericality alone, but `validates :price, numericality: { greater_than: 0 }`
  allows `nil` by default (it skips validation when the value is blank).

## SQLite notes

No SQLite-specific limitations for this assignment. Uniqueness is enforced at the
model level (not just via the DB index), and all standard validations work normally.
